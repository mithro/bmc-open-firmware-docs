# Tim's Random Documentation for his Open BMC & Firmware

Sphinx/MyST source for a documentation site about replacing the proprietary
management firmware on three pieces of legacy hardware with fully open
alternatives: open drivers, open bootloaders, open BMC stacks, and QEMU
emulation faithful enough to develop all of it against.

**Rendered site:** <https://bmc-open-firmware-docs.readthedocs.io/>

Every pushed branch is published as its own live Read the Docs version, and
pull requests get a preview build linked from their status checks.

> [!WARNING]
> This repository has been primarily generated by AI (Claude). Its content has
> been cross-referenced against multiple independent sources — vendor
> datasheets, reverse engineering of the shipped firmware, schematic tracing,
> and mainline kernel/coreboot sources — and validated against real silicon
> where the pages say so. Verification status varies by page: each page carries
> its own citations and evidence, and claims proven on physical hardware are
> marked as such. Treat unmarked claims as documented-but-not-yet-measured.

## One body of hardware truth, three consumers

The site is built around a single idea. Document the hardware once, completely
and accurately, and three kinds of work can all be driven from that one
reference:

1. **Drivers.** Enough register-level and board-level detail to write drivers
   for U-Boot, Zephyr, and Linux, plus the userspace tooling above them.
2. **Emulation.** Enough behavioural detail to build hardware-accurate QEMU
   models — accurate enough to develop the drivers from (1) against, to
   replicate the real part's hard-to-find bugs, and to boot the unmodified
   proprietary firmware that shipped with each board.
3. **Verification.** Small, self-contained tests that run identically against
   QEMU and against the real silicon, so the model and the chip are proven to
   agree rather than assumed to.

Around that core, the docs also do three bookkeeping jobs: track the status of
the driver, emulation, and test work with links to the code; collect reference
material cross-checked across as many independent sources as possible; and
record what has actually been measured on the physical boards, kept distinct
from what was merely inferred from documentation.

## The hardware

### ASUS KGPE-D16

A dual-socket AMD Opteron server motherboard. Its BMC is an **Aspeed AST2050**
(also sold as AST1100): an ARM926EJ-S at 200 MHz with 64 MiB of DDR2. The
AST2050 is not supported by mainline Linux; the earliest mainline Aspeed
generation is the AST2400 (G4). This board is the primary development and
silicon-validation target, backed by a fully remote-controllable hardware rig:
JTAG, serial console, a PCIe-to-AHB (P2A) bridge, switched power, and video
capture. See [systems/kgpe-d16](https://bmc-open-firmware-docs.readthedocs.io/en/latest/systems/kgpe-d16.html).

### Dell PowerEdge C410X

A 3U, 16-slot PCIe GPU expansion chassis with no host CPU; it is managed
entirely by the same AST2050 SoC, running Avocent MergePoint firmware. It
contributes the vendor-firmware reverse engineering and a second board profile
(I2C topology, GPIO map, power sequencing) on top of the shared SoC work. See
[systems/dell-c410x](https://bmc-open-firmware-docs.readthedocs.io/en/latest/systems/dell-c410x.html).

### HPE Intelligent Modular PDU (AF531A)

A different SoC entirely: a **Digi NS9360**, also ARM926EJ-S. Its stock
firmware is NET+OS (a ThreadX RTOS), not Linux, and there is no mainline Linux
support for the chip. Its open-firmware path therefore starts with a U-Boot
port. See [systems/hpe-ipdu](https://bmc-open-firmware-docs.readthedocs.io/en/latest/systems/hpe-ipdu.html).

Two of the three boards share the AST2050, so the SoC-level work (drivers,
U-Boot, QEMU model) is shared and each board adds only its own device tree and
board description. The NS9360 is a separate track on the same CPU core.

## Status, by consumer

Each track's page carries its status and links to the code.

**Drivers**

- [drivers/linux](https://bmc-open-firmware-docs.readthedocs.io/en/latest/drivers/linux.html) — the AST2050 (G3) patch series on a modern kernel, including two silicon-verified fixes: the G3 VIC interrupt controller and the ftgmac100 `FAST_MODE` RX bug.
- [drivers/uboot](https://bmc-open-firmware-docs.readthedocs.io/en/latest/drivers/uboot.html) — the AST2050 and NS9360 U-Boot ports and their netboot paths.
- [drivers/zephyr](https://bmc-open-firmware-docs.readthedocs.io/en/latest/drivers/zephyr.html) — the ARMv5/ARM926EJ-S architecture port that the Zephyr track requires.
- [drivers/peripheral-map](https://bmc-open-firmware-docs.readthedocs.io/en/latest/drivers/peripheral-map.html) — the cross-reference from every hardware block to its Linux/U-Boot/Zephyr driver and QEMU model.

**Emulation**

- [emulation/qemu](https://bmc-open-firmware-docs.readthedocs.io/en/latest/emulation/qemu.html) — the new `kgpe-d16-bmc` (AST2050) and `ns9360` QEMU machines, the faithful per-peripheral G3 model, and their boot criteria.
- [emulation/testbench](https://bmc-open-firmware-docs.readthedocs.io/en/latest/emulation/testbench.html) — qtest register-level benches plus a `firmware-testbench` layer whose same tests run against QEMU in CI and against real boards on the bench.

**Verification on silicon**

- [debug/bring-up](https://bmc-open-firmware-docs.readthedocs.io/en/latest/debug/bring-up.html) — cold-booting a firmware-less AST2050 from the host over the P2A bridge; everything on the page is hardware-verified.
- [debug/jtag-uart](https://bmc-open-firmware-docs.readthedocs.io/en/latest/debug/jtag-uart.html) — JTAG run-control on the real chip, cross-checked against the P2A path.
- [firmware/openbmc](https://bmc-open-firmware-docs.readthedocs.io/en/latest/firmware/openbmc.html) — the per-feature table that records honestly which BMC features are proven on silicon and which only in QEMU.

### Evidence the method works

- The AST2050 QEMU machine boots three independent firmware stacks: the
  program's own mainline-based U-Boot + Linux to SSH, the vintage Raptor
  Engineering 2.6.28.9 AST2050 kernel to SSH, and the unmodified proprietary
  Dell C410X firmware all the way to its serving web interface
  ([emulation/qemu](https://bmc-open-firmware-docs.readthedocs.io/en/latest/emulation/qemu.html)).
- A stripped, Redfish-only modern OpenBMC has booted on the real 64 MiB
  AST2050 over NFS and answered Redfish v1.17.0 over HTTPS
  ([firmware/openbmc](https://bmc-open-firmware-docs.readthedocs.io/en/latest/firmware/openbmc.html)).
- JTAG run-control and in-band access both work on the real chip, and the two
  independent paths read the same SoC revision register
  ([debug/jtag-uart](https://bmc-open-firmware-docs.readthedocs.io/en/latest/debug/jtag-uart.html),
  [debug/bring-up](https://bmc-open-firmware-docs.readthedocs.io/en/latest/debug/bring-up.html)).
- The NS9360 U-Boot port boots under a custom QEMU `ns9360` machine model
  ([drivers/uboot](https://bmc-open-firmware-docs.readthedocs.io/en/latest/drivers/uboot.html)).

## Building locally

```sh
uv run --with-requirements requirements.txt sphinx-build -M html source _build -W --keep-going
```

Warnings are errors. CI runs the same build plus an external link check. Open
`_build/html/index.html` when it finishes.
