# Nuvoton W83795G / W83795ADG hardware monitor

## 1.1 Overview and the two variants

The W83795G is a member of Nuvoton's (formerly Winbond) hardware-monitor family.
It integrates a **10-bit ADC** and can simultaneously monitor system voltages,
fan speeds and temperatures, drive fan PWM/DC outputs with Smart-Fan control,
and act as an **ASF 2.0** sensor for out-of-band management. [W83795G DS p.4](#sources)

Two package/feature variants share one datasheet and one register model:

## 1.4 The banked register model

The device exposes **four register banks (Bank 0–3)** through a single 8-bit
address space. Software must first write the **Bank Select** register at address
`00` in any bank, then read/write the target register. Address `00` is the Bank
Select register in *all four* banks, so it is always reachable.
[W83795G DS p.28,30-31](#sources)

```{list-table} Bank Select register (address 00, all banks) — RW, POR 0x80
:header-rows: 1
:widths: 16 26 58

* - Bit
  - Name
  - Description
* - 7
  - HBACS
  - High-byte access: 0 = read low byte of the 2-byte Vendor ID; 1 = read high byte. [W83795G DS p.30](#sources)
* - 6-3
  - Reserved
  - Reserved.
* - 2-0
  - Bank Select
  - `000`=Bank0, `001`=Bank1, `010`=Bank2, `011`=Bank3. [W83795G DS p.30](#sources)
```

Identification registers (readable in any bank): **Vendor ID** `CRFD` =
`0x5C`/`0xA3` (high/low, selected by HBACS), **Chip ID** `CRFE` = `0x79`,
**Device ID** `CRFB` = `0x51` (B-version) / `0x52` (C-version).
[W83795G DS p.30-31](#sources)

## 1.6 Monitoring value formats and scaling

**Voltage** — Each channel readout is a 10-bit count: high byte in the channel's
reading register (bits [9:2]) plus **VR LSB** (`CR3C`, bits [1:0]). Reading the
high byte latches VR LSB for the next read. [W83795G DS p.44](#sources)

```{list-table} Voltage scaling (10-bit count → volts)
:header-rows: 1
:widths: 55 45

* - Channels
  - Formula
* - VSEN1–VSEN13, VDSEN14–VDSEN17, VTT
  - V = count × 0.002 (2 mV / LSB)
* - 3VSB, 3VDD, VBAT
  - V = count × 0.006 (6 mV / LSB)
```

Source: [W83795G DS p.44](#sources). Cross-check: the Linux [`w83795`](https://github.com/torvalds/linux/blob/master/drivers/hwmon/w83795.c) driver applies the
same 2 mV/6 mV scaling internally. [kernel.org w83795.html](#sources) For VSEN14–VSEN17 an
optional **×8 input gain buffer** (VMIGB CTRL, §1.7) changes the effective
range.

**Temperature** — 10-bit two's complement (9-bit + sign): high byte bits [9:2] =
whole degrees with sign, VR LSB bits [1:0] = 0.5 °C and 0.25 °C fractions. Range
−128 °C … +127.75 °C; a reading of **−128 °C means the thermal diode is open**.
[W83795G DS p.45,47](#sources)

**Fan tachometer** — 12-bit count: FANIN_COUNT high byte [11:4] + VR LSB [3:0].
[W83795G DS p.48](#sources)

```{admonition} Fan RPM formula
:class: note

$$
\text{FanSpeed (RPM)} = \frac{1.35 \times 10^6}{\text{count} \times (\text{FanPoles} / 4)}
$$

FanPoles = number of N-S pole pairs; a normal fan (FanPoles = 4) makes 2 pulses
per revolution. Tachometer frequency must stay below 4.5 kHz (135 kRPM at
FanPoles=4). [W83795G DS p.49](#sources)
```

## 1.7.3 Watchdog timer

The W83795G/ADG has a watchdog that pulses **WDTRST#** (pin 39) on timeout.
There are **soft** and **hard** variants. Enable via the WDT LOCK unlock codes
(`55`/`33`), confirm via WDT ENABLE, program timeout in WDT TIMER (minutes for
soft; write `00` to disable in hard mode). WDT STS records the last stage for BIOS
readout. [W83795G DS p.40-43](#sources) This is directly usable as a **board-level system
reset** independent of the BMC.

## 1.9 VID / AMD Opteron voltage tracking

VSEN1/2/3 track the CPU/NB core rails and can auto-update their limits from VID.
The **VID_SEL[2:0]** field (VID CTRL, `CR6A`) selects the VID table — crucially
for the KGPE-D16, it supports **AMD Opteron 5-bit and 6-bit parallel VID** and
**AMD SVID (serial VID)** on the SCL_SVI/SDA_SVI pins: [W83795G DS p.67](#sources)

```{list-table} VID_SEL (VID CTRL 6A, bits 2-0)

:widths: 18 82
```

## 1.10.1 The three Smart-Fan modes

```{admonition} Fan control modes
:class: note

- **Manual mode** — FANCTL not mapped to any temperature; software writes FOV
  directly (PWM duty or DC level). [W83795G DS p.104](#sources)
- **Speed Cruise** (FCMS1 per-FANCTL) — closed-loop on **FANIN tachometer**: the
  chip tunes the output to hold each fan at its FTS target ±TFTS. FANCTLn maps
  1:1 to FANINn. [W83795G DS p.95](#sources)
- **Thermal Cruise** (FCMS2 bit = 0) — closed-loop on **temperature**: holds a
  temperature source near its TTTI target within ±HT; below the low end the fan
  drops to FONV then stops after FOST; above the high end it ramps up, and hits
  full speed / warns if it still can't hold. [W83795G DS p.96-97](#sources)
- **SMART FAN IV** (FCMS2 bit = 1) — a **6-slope temperature→duty curve** per
  temperature channel, defined by 7 setpoints (T1–T7) and 7 outputs
  (DC/PWM1–DC/PWM7) in the SFIV block; above CTFS/critical it forces DC/PWM7 /
  full speed. [W83795G DS p.98,113](#sources)
```

**SMART FAN IV table layout** (SFIV, `80`–`DE`): six tables (one per temperature
source), each with a 7-entry temperature row (`T1–T7`) and a 7-entry output row
(`DC/PWM1–7`). Between setpoints the output is linearly interpolated; the slope
between points 4 and 5, for example, is $X4 = \dfrac{\text{DC/PWM5} - \text{DC/PWM4}}{T5 - T4}$.
[W83795G DS p.98,113](#sources)

Source: [W83795G DS p.113](#sources). Temperature sources are chosen with the **TSS**

## 1.13 How the KGPE-D16 uses the W83795G

- **Placement**: the board's shared platform sensor bus (`QU4`, SCL/SDA pins
  33/34) — one wire driven **multi-master** by the host SP5100 (`SMBus1`,
  `i2c-piix4`, I/O `0x0B00`) *and* the AST2050 BMC (`I2C2`, balls C14/D14);
  7-bit `0x2F`; enumerates in host Linux as `w83795g-i2c-1-2f` on `hwmon`.
  [hardware-inventory/README.md:42-44](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/hardware-inventory/README.md#L42-L44) [docs.dasharo.com fan-control](#sources)
  [I2C-SMBUS-TOPOLOGY.md](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/I2C-SMBUS-TOPOLOGY.md#32-shared-platform-sensor-bus-multi-master)
- **Other board wiring** (from the schematic netlist): the TSI pins 29/30
  connect to the BMC's `I2C4` through level-shift FETs `Q56`–`Q59` (the
  CPU-thermal path — {doc}`/systems/kgpe-d16-i2c` §4.4); pin 38 `PWRBTN#`
  shares the `AST_PWRBTN#` net with the BMC and front panel; its reset comes
  from the Super-I/O's `RSTOUT1#`; its clock is the SP5100's `PCICLK3`.
  [I2C-SMBUS-TOPOLOGY.md](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/I2C-SMBUS-TOPOLOGY.md#34-cpu-thermal-sb-tsi)
  [W83667HG-SUPERIO-WIRING.md](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/W83667HG-SUPERIO-WIRING.md#6-power--reset--acpi-sequencing)
  [SP5100-SOUTHBRIDGE-WIRING.md](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/SP5100-SOUTHBRIDGE-WIRING.md#5-pci-33-mhz-host--slots--bmc-vgaikvm)
- **Fan zones**: two — **CPUFAN_SEL1** (CPU fans) and **CHAFAN_SEL1** (chassis
  fans), each assignable to a 4-pin (PWM) or 3-pin (DC/voltage) FANCTL output.
  [docs.dasharo.com fan-control](#sources) On the observed board `fan1` reads ~2600 RPM and
  the other fan headers are unpopulated. [hardware-inventory/README.md:44](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/hardware-inventory/README.md#L44)
- **Fan control**: coreboot (Dasharo ≥ v0.2.0) uses **Thermal Cruise**, target
  ~50 °C, critical 90 °C → full speed, `pwm1` (0–255), `fan1` min ~329 RPM.
  [docs.dasharo.com fan-control](#sources)
- **Temperature sources**: CPU via **AMD SB-TSI** (Bank 3, `temp7_input`) plus a
  thermal-diode/thermistor input (`temp1`); NB temperature is also read for the
  chassis-fan zone. [docs.dasharo.com fan-control](#sources)
- **CPU voltage**: VSEN1/2/3 track VCORE/VDDNB via AMD VID/SVI (§1.9).
- **BMC access**: the AST2050 **can** drive this chip directly — its `I2C2`
  controller sits on the same shared sensor bus (the earlier claim here that
  only the host could reach it predated the schematic tracing). An open BMC
  firmware must respect the multi-master arbitration hazards documented in
  {doc}`/systems/kgpe-d16-i2c` §3: the host owns the bus during POST and
  runtime, so out-of-band reads are safest with the host off or coordinated.
