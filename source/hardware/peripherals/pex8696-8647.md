# PEX8696 / PEX8647 — PCIe switches (I2C management)

## 3.1 Device parameters and roles

```{list-table} PLX/Broadcom PEX switches on the C410X [PEX-I2C], [Broadcom-8696], [Broadcom-8647]
:header-rows: 1
:widths: 16 10 44 30

* - Part
  - Qty
  - Description
  - Role
* - PEX8696
  - 4
  - 96-lane, 24-port PCIe **Gen2** (5.0 GT/s) switch; 24 ports grouped into 6 stations of 4 ports
  - **Downstream** — fan out to the 16 GPU slots, 4 slots per switch
* - PEX8647
  - 2
  - 48-lane, 3-port (3×x16) PCIe Gen2 switch
  - **Upstream** — host-side links from the iPass cables
```

All six switches sit on **I2C bus 3** of the AST2050 with **no mux**; the firmware
encodes bus+mux in one byte `0xF3` (low nibble `3` = bus 3, high nibble `F` = no
mux) and reaches them through `/dev/aess_i2cdrv` [PEX-I2C], [pex-README].

The 8-bit form is the 7-bit address left-shifted by one (GBT format); the kernel
driver inserts the R/W bit [PEX-I2C].

## 3.2 Family-level architecture (why an I2C control path exists)

Across the ExpressLane 86xx Gen2 family, **every port has a hot-plug controller
and a hot-plug control/status register that is reachable "through external logic
via the I2C interface"**, and the switch can be configured "with strapping pins,
EEPROM, I2C, or Host software" [PEX8696 PB], [PEX8647 PB]. Multi-host operation is
built on a **Non-Transparent (NT) bridge port** that isolates host memory domains
while allowing inter-host communication through base-address / doorbell / scratchpad
registers [PEX8696 PB], [PEX8647 PB]. The SerDes PHY has **software-controlled power
and signal levels** (four levels: off/low/typical/high) [PEX8696 PB]. Interrupts
(INTx or MSI) are raised for **hot-plug events, doorbells, and error reporting**
[PEX8696 PB], [PEX8647 PB]. The C410X BMC drives exactly these facilities over I2C.

## 3.3 The PLX 4-byte I2C command protocol

The PEX8696/PEX8647 do **not** use a simple register-pointer scheme. Every config
access is framed as a **4-byte command header** that carries a command code, a
station/port selector, a byte-enable mask, and a 10-bit DWORD register index.

```
4-byte command:  [cmd] [stn_port] [enables|reg_hi] [reg_lo]
```

```{list-table} Command header byte fields [PEX-I2C]
:header-rows: 1
:widths: 12 30 58

* - Byte
  - Field
  - Encoding
* - `[0]`
  - Command
  - `0x03` = write (send 4 data bytes to the register); `0x04` = read (read 4 data bytes back)
* - `[1]`
  - Station / port
  - bits 7:1 = `(station << 1) | (port >> 1)`; bit 0 = 0
* - `[2]`
  - Enables + reg-hi + port-lo
  - bit 7 = `port & 1` (low bit of port); bits 5:2 = byte-enable mask (`0xF` = all 4 bytes); bits 1:0 = register DWORD index bits [9:8]
* - `[3]`
  - reg-lo
  - register DWORD index bits [7:0]
```

- **Register DWORD index** = `((byte[2] & 0x03) << 8) | byte[3]`; the **byte
  address** = index × 4 [PEX-I2C].
- In the C410X firmware `byte[2]` is almost always **`0x3C`** (enables = all four
  bytes, reg-hi = 0, port-lo = 0), so the DWORD index reduces to `byte[3]`
  [PEX-I2C]. For odd ports it becomes `0xBC` (bit 7 set); for a DWORD index ≥ 0x100
  the reg-hi bits are non-zero (e.g. SerDes index `0x2E4` uses `byte[2] = 0x3E`,
  `byte[3] = 0xE4`) [PEX-I2C].
- **Values are 32-bit little-endian**: data byte 0 = value bits [7:0] … data byte 3
  = value bits [31:24], on both read and write [PEX-I2C].

**Write** transfers 8 bytes on the wire (4 command + 4 value); **read** is a
write-then-repeated-start-read (4 command bytes, repeated START, 4 value bytes).

```
Write:  START | addr+W | 03 | stn_port | enables | reg_lo | v0 v1 v2 v3 | STOP
Read :  START | addr+W | 04 | stn_port | enables | reg_lo |
        repeated START | addr+R | v0 v1 v2 v3 | STOP
```

**Station/port addressing.** The PEX8696's 24 ports are 6 stations × 4 ports; the
firmware pre-computes `byte[1]` per slot. Each PEX8696 uses stations 1, 2, 4, 5 for
its four GPU slots (global ports 4, 8, 16, 20); station 0 is the upstream/config
port and station 3 port 3 (global port 15) is the NT-bridge port [PEX-I2C].

**7-bit vs 8-bit addresses.** PEX8696 = 7-bit `0x18/0x19/0x1A/0x1B` (8-bit
`0x30/0x32/0x34/0x36`); PEX8647 = 7-bit `0x68/0x6A` (8-bit `0xD0/0xD4`) [PEX-I2C].

**Contrast with the plxtools 2-byte protocol.** Smaller PLX parts (and the
`plxtools` I2C backend) use a flat **2-byte, big-endian** register address with
implicit 32-bit access and no explicit byte-enables; the 4-byte command form above
is required for the PEX8696 because per-port register access must encode
station/port and byte-enables in the command [PEX-I2C].

```{list-table} 2-byte (plxtools) vs 4-byte (Dell) PLX I2C protocol [PEX-I2C]
:header-rows: 1
:widths: 26 34 40

* - Feature
  - plxtools (2-byte addr)
  - Dell firmware (4-byte cmd)
* - Address form
  - 2 bytes, big-endian
  - 4 bytes (cmd, stn/port, enables, reg)
* - Port selection
  - via BAR offset
  - encoded in bytes [1] and [2]
* - Register offset
  - 16-bit flat
  - 10-bit DWORD index, per port
* - Byte enables
  - implicit (always 32-bit)
  - explicit in byte[2] bits 5:2
```

## 3.4 Register reference

Config addresses live in each switch's per-port space. The PCIe Express Capability
base is offset **`0x068`** in the PEX8500/8600/8696 series, so Slot Capabilities =
`0x068 + 0x14 = 0x07C` and Slot Control/Status = `0x068 + 0x18 = 0x080` [PEX-I2C].

(SerDes DWORD indices ≥ `0x100` set the reg-hi bits in byte[2]; e.g. `0xB90` =
DWORD `0x2E4`, sent as `byte[2]=0x3E`, `byte[3]=0xE4` [PEX-I2C].)

**Register `0x07C` — Slot Capabilities / write-protect.** Standard PCIe read-only
slot-capability fields, but PLX repurposes **bit 18** as a **write-protect enable**
writable over I2C: when set, the VS1 register block (`0x200+`) is protected. It must
be **cleared** before modifying any hot-plug or lane register [PEX-I2C].

**Register `0x080` — Slot Control (low 16b) / Slot Status (high 16b).** Standard
PCIe hot-plug register. The firmware manipulates **PIC** (Power Indicator Control,
bits 9:8), **PCC** (Power Controller Control, bit 10; **0 = power on, 1 = power
off**) and **AIC** (Attention Indicator Control, bits 7:6). Indicator codes:
`01 = on`, `10 = blink`, `11 = off` [PEX-I2C].

**Register `0x234` — hot-plug power controller.** **Bit 0** is *pulsed*
(assert → wait 100 ms → de-assert) to trigger the PLX hardware power controller
that sequences a slot's rails with inrush control; on the PEX8647, **bit 8**
selects the primary vs secondary upstream host port for multi-host modes [PEX-I2C].

**Register `0x228` — hot-plug LED / MRL.** **Bit 21** is set (never cleared) during
power-on to enable the hardware hot-plug LED for the port [PEX-I2C].

**Registers `0x380`/`0x384` — lane config**, and **`0x3AC` — NT bridge**: written
to the config port (station 0) and NT port (station 3 port 3) to reconfigure the
crossbar for multi-host modes [PEX-I2C]. **`0x1DC` bit 19** on the PEX8647 merges
its two upstream ports into a wider aggregated link (8:1 mode) [PEX-I2C].

## 3.6 Per-slot power-on sequence (9 I2C transactions + 100 ms)

For each slot (firmware `pex8696_slot_power_on_reg`), after checking PSU power-good
and disabling the hot-plug IRQ [PEX-I2C]:

```{list-table} Power-on transaction sequence [PEX-I2C]
:header-rows: 1
:widths: 8 14 20 58

* - Step
  - Register
  - Op
  - Action
* - 1
  - `0x07C`
  - read+write
  - Clear write-protect **bit 18** (`byte[2] of value &= 0xFB`)
* - 2
  - `0x080`
  - read+write
  - Set **PIC = 01 (on)**, clear **PCC (bit 10 = 0 → power on)** — `v1 = (v1 & 0xFC | 0x01) & 0xFB`
* - 3
  - `0x234`
  - read+write
  - **Assert bit 0** (`v0 |= 0x01`) — start hardware power controller
* - 4
  - —
  - delay
  - **Sleep 100 ms** (10 RTOS ticks)
* - 5
  - `0x234`
  - write
  - **De-assert bit 0** (`v0 &= 0xFE`) — end the pulse
* - 6
  - `0x228`
  - read+write
  - Set **bit 21** (`v2 |= 0x20`) — enable hot-plug LED
```

Per slot: 4 reads + 5 writes = **9 transactions** plus one 100 ms delay [PEX-I2C].

## 3.7 Per-slot power-off (1 write)

Power-off is a single write to `0x080` setting the low bits `| 0x07`: **PIC = 11
(off)** and **PCC = 1 (power off)**; no `0x234` pulse, no `0x228`, no write-protect
removal (`0x080` is a standard PCIe register) [PEX-I2C]. All-slots-off is either 32
read-modify-write transactions or 16 pre-built writes [PEX-I2C].

## 3.8 Staggered full-chassis power-on

The orchestrator `Start_GPU_Power_Sequence` powers all 16 slots in **4 phases,
one slot per PEX8696 switch per phase**, to spread inrush current [PEX-I2C]:

```{list-table} 4-phase GPU power-on [PEX-I2C]
:header-rows: 1
:widths: 12 30 34 24

* - Phase
  - Function
  - Slots (1-based)
  - `gpu_un_protect` mask
* - 1
  - `gpu_power_on_4_8_12_16`
  - 4, 8, 12, 16
  - `0xFF`
* - 2
  - `gpu_power_on_3_7_11_15`
  - 3, 7, 11, 15
  - `0x77`
* - 3
  - `gpu_power_on_2_6_10_14`
  - 2, 6, 10, 14
  - `0x33`
* - 4
  - `gpu_power_on_1_5_9_13`
  - 1, 5, 9, 13
  - `0x11`
```

Totals: 36 PEX transactions and 4 × 100 ms per phase → **144 transactions, 16
delays, ~2.0 s** for a full 16-slot boot; `filter_on_gpu()` skips empty slots
[PEX-I2C].

## 3.9 Multi-host modes (2:1 / 4:1 / 8:1)

The chassis supports three host-to-GPU fan-out ratios, switched via IPMI. Mode
switching (orchestrator at `0x38A98`) powers all slots off, rewrites PEX8696 lane
config (`0x380`/`0x384`), rewrites PEX8647 port config (`0x234`/`0x1DC` with 200 ms
delays), sets the NT bridge (`0x3AC` on port 15), then re-equalises SerDes on every
port (`0xB90`/`0xB9C`/`0xBA4`/`0xBA8`/`0x204`) — ~153 I2C transactions total
[PEX-I2C].

## 3.10 EEPROM access

Each switch has an attached serial EEPROM reached through the PLX EEPROM-controller
registers at DWORD offsets **`0x260`** (control) and **`0x264`** (data), using the
same 4-byte command protocol; the firmware wraps this asynchronously via message
queues (`read_plx_eeprom` / `write_plx_eeprom`) and stores 8-byte serial numbers
there [PEX-I2C].

## 3.11 Model / re-implementation notes

A replacement BMC only needs I2C bus 3 (no mux), the slot map (§3.5), and the
`plx_read`/`plx_write` helpers with the 4-byte header (§3.3) to power slots on/off.
This is the heaviest peripheral to model in QEMU/OpenBMC: scope the model to the
command framing plus the ~13 config DWORDs the firmware actually touches and the
EEPROM shadow [PEX-I2C], [pex-README].
