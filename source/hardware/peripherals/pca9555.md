# PCA9555 — 16-bit I2C GPIO expander

The PCA9555 is a "remote 16-bit I2C and SMBus I/O expander with interrupt output
and configuration registers" [PCA9555 DS p.1](#sources). It presents two 8-bit ports
(Port 0 = P0.0–P0.7, Port 1 = P1.0–P1.7) over a two-wire I2C interface, each pin
independently configurable as input or output, with an open-drain active-low
`INT` output and a polarity-inversion feature on inputs [PCA9555 DS p.1](#sources),
[pca953x](#sources). The five instances contribute 80 of the board's 118 GPIO lines
[gpio-map](#sources), [io-tables](#sources).

## 1.1 Device address and the 8 A2/A1/A0 instances

The 7-bit slave address is a fixed prefix `0100` followed by the three hardware
address pins A2,A1,A0, giving eight addresses `0x20`–`0x27`; the R/W bit is
appended as the 8th bit of the transmitted address byte [PCA9555 DS p.19](#sources).

(see §1.5) [gpio-map](#sources), [io-tables](#sources).

## 1.2 Command byte and register map (8 registers)

After the address byte, the master sends a **command byte** whose low 3 bits
(B2,B1,B0; bits 7:3 = 0) select one of the eight internal registers; that register
then stays selected for subsequent reads until a new command byte is written
[PCA9555 DS p.19](#sources). The registers are organised as four register **pairs** (Port 0
even, Port 1 odd); after writing one register of a pair the next data byte lands
in the other register of the pair, so both ports can be updated in one transaction
[PCA9555 DS p.21](#sources).

```{list-table} PCA9555 register map (command byte 0x00–0x07) [PCA9555 DS p.19](#sources), [PCA9555 DS p.20](#sources)
:header-rows: 1
:widths: 12 26 18 20 24

* - Cmd
  - Register
  - Access
  - POR default
  - Function
* - `0x00`
  - Input Port 0
  - Read only
  - `xxxxxxxx`
  - Live logic level of P0.x pins (after optional polarity invert); writes ignored
* - `0x01`
  - Input Port 1
  - Read only
  - `xxxxxxxx`
  - Live logic level of P1.x pins; writes ignored
* - `0x02`
  - Output Port 0
  - Read/write
  - `11111111`
  - Value driven onto P0.x pins configured as outputs (reads the FF, not the pin)
* - `0x03`
  - Output Port 1
  - Read/write
  - `11111111`
  - Value driven onto P1.x pins configured as outputs
* - `0x04`
  - Polarity Inversion 0
  - Read/write
  - `00000000`
  - 1 = invert the corresponding P0.x bit as read back in Input Port 0
* - `0x05`
  - Polarity Inversion 1
  - Read/write
  - `00000000`
  - 1 = invert the corresponding P1.x bit as read back in Input Port 1
* - `0x06`
  - Configuration 0
  - Read/write
  - `11111111`
  - Direction of P0.x: **1 = input** (hi-Z), **0 = output**
* - `0x07`
  - Configuration 1
  - Read/write
  - `11111111`
  - Direction of P1.x: 1 = input, 0 = output
```

Semantics [PCA9555 DS p.20]:

- **Input Port (0x00/0x01)** reflect the incoming pin levels regardless of
  direction; read-only; a write has no effect.
- **Output Port (0x02/0x03)** hold the drive value for output pins; reading them
  returns the flip-flop contents, *not* the actual pin.
- **Polarity Inversion (0x04/0x05)** invert how *input* pins appear in the Input
  Port register (used on active-low signals so software reads "1 = asserted").
- **Configuration (0x06/0x07)** set direction; reset = all inputs.

## 1.3 Power-on reset and output drive

An internal POR holds the device in reset until `VCC` reaches `VPORR`
(typ 1.2 V, max 1.5 V); to force a fresh POR, `VCC` must fall below `VPORF`
(typ 1 V, min 0.75 V) then rise again [PCA9555 DS p.7](#sources), [PCA9555 DS p.15](#sources). At
reset all I/Os are inputs and registers take the defaults in the table above
[PCA9555 DS p.14](#sources), [PCA9555 DS p.20](#sources). Outputs are **latched, high-current**
drivers able to sink LEDs directly; an input pin has an internal ~100 kΩ pull-up
to `VCC`, and all I/O is 5-V tolerant [PCA9555 DS p.1](#sources), [PCA9555 DS p.14](#sources).

## 1.4 Interrupt (`INT`) behaviour and errata

`INT` is open-drain (needs an external pull-up) and asserts low whenever any input
pin's state differs from what is stored in the Input Port register; it is cleared
when the changed port is read, or when the pin returns to its stored value. Port 0
and Port 1 latch and clear **independently** — reading Port 0 does not clear a
Port-1-caused interrupt [PCA9555 DS p.16](#sources). A pin configured as an output cannot
raise `INT`; changing a pin from output to input can produce a false interrupt if
its level does not match the stored Input value [PCA9555 DS p.16](#sources).

```{admonition} Interrupt errata (matters for a re-implementation)
:class: warning

`INT` can be **improperly de-asserted** if *both*: (1) the last command byte
written was `0x00` (Input Port 0 — the usual state after reading inputs), and
(2) another slave on the bus ACKs an address byte with the R/W bit high. The
master then sees the interrupt as spuriously cleared. Workaround: after a read of
a PCA9555, write a command byte other than `0x00` before addressing any other
slave [PCA9555 DS p.16](#sources).
```

## 1.5 The C410X's five PCA9555 instances

```{list-table} C410X PCA9555 instances [gpio-map](#sources), [io-tables](#sources)
:header-rows: 1
:widths: 8 10 12 12 34 24

* - #
  - Bus
  - 7-bit
  - 8-bit
  - Role
  - `INT` → AST2050 pin
* - 1
  - `0xF6`
  - `0x20`
  - `0x44`*
  - GPU card presence-detect, 16 slots (PRSNT#, active-low, inverted in SW)
  - GPIOA4
* - 2
  - `0xF6`
  - `0x21`
  - `0x42`
  - Per-slot 12 V power-good feedback, 16 slots (inputs, polarity-inverted)
  - GPIOB5
* - 3
  - `0xF6`
  - `0x22`
  - `0x44`
  - Per-slot attention buttons (inputs) + slot power-enable control (outputs)
  - GPIOB4
* - 4
  - `0xF6`
  - `0x23`
  - `0x46`
  - Per-slot MRL (manual-retention-latch) sensors, 16 inputs
  - GPIOB5 (shared with #2)
* - 5
  - `0xF1`
  - `0x20`
  - `0x40`
  - PSU present/fail (P0.0–3 in), PS_ON# control bit (P0.4 out), fan-status LEDs (Port 1 out)
  - GPIOA5
```

\* Presence-detect PCA9555 #1 is documented as read via 8-bit address `0x44` in
`read_gpu_present()` and attention PCA9555 #3 also via `0x44`; the two live on
different buses (both `0xF6` here) — the firmware's decompiled accessors are the
authority for the exact byte used [gpio-map](#sources). The 7-bit strap addresses `0x20`
and `0x22` are the datasheet-consistent values.

Notable pin assignments (full 80-pin map in [gpio-map](#sources)):

- **#1 @ 0xF6:0x20** — P0.0–P0.7 = slots 1–8 PRSNT#, P1.0–P1.7 = slots 9–16
  PRSNT#. `INT` on GPIOA4 fires on any insert/remove; the handler reads both
  ports, diffs against the previous state, logs plug/unplug to the SEL, and
  power-offs a hot-removed powered slot [gpio-map](#sources).
- **#3 @ 0xF6:0x22** — Port bits map to the 16 per-slot attention buttons; a
  single-button press (multi-press rejected) triggers that slot's hot-plug
  power sequence [gpio-map](#sources).
- **#5 @ 0xF1:0x20** — P0.4 is the **PS_ON# control bit**: `ps_on_pull_down()`
  clears it (AND `~0x10`) to turn all PSUs on, `ps_on_pull_up()` sets it
  (OR `0x10`) to turn them off; Port 1 drives the 8 bicolor fan-status LEDs
  [gpio-map](#sources).

## 1.6 Linux binding (`pca953x`)

Mainline Linux drives the PCA9555 with **`drivers/gpio/gpio-pca953x.c`**, DT
compatible **`nxp,pca9555`** (16 lines, interrupt-capable) [pca953x](#sources). The node
provides `gpio-controller` + `#gpio-cells = <2>`, and — because each device's
`INT` wires to an AST2050 GPIO — `interrupt-controller` + `#interrupt-cells = <2>`
plus an `interrupts` property. For the fixed-function outputs that implement the
power sequence (e.g. the PS_ON# bit) a reconstructed device tree would name the
lines and use GPIO **hogs** [pca953x](#sources).
