# W83601G — 15-bit I2C/SMBus GPIO expander

The Winbond (now Nuvoton) W83601G is a "general purpose input/output IC with
SMBus (I2C)" providing **15 GPI/O pins** in two ports, an interrupt output that
signals transitions on inputs, and a programmable blinking "Power LED" mode
[W83601G DS p.2](#sources). It speaks the SMBus 1.0 Write Byte / Read Byte
protocol at 3.3 V signal levels and comes in a 20-pin SSOP package
[W83601G DS p.2](#sources). The ASUS KGPE-D16 populates **two** of them — `U27`
and `U28`, identified by the schematic part-description field
"C.S W83601G SSOP20//WINBOND MULTI-FUNCTION G.P.I/O"
[QU1_pins.md:256-257](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/pinmaps/QU1_pins.md#L256-L257) —
on the AST2050 BMC's I2C5 inventory bus, where they drive the board's **16
per-DIMM-slot error LEDs**
[AST2050-BMC-WIRING.md:353-354](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/AST2050-BMC-WIRING.md#L353-L354).

## 1.1 The two KGPE-D16 instances: DIMM error LEDs

Both expanders sit directly (no mux) on the BMC's **I2C5** — AST2050 `QU1`
balls **A13** (`SDA5/GPIOC6`, net `N39526313`) and **B13** (`SCL5/GPIOC7`, net
`N39526322`) — sharing the segment with the board FRU EEPROM, a Holtek HT24LC08
({doc}`ht24lc08`, `U25`)
[QU1_pins.md:262-266](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/pinmaps/QU1_pins.md#L262-L266),
[I2C-SMBUS-TOPOLOGY.md:155-156](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/I2C-SMBUS-TOPOLOGY.md#L155-L156).
The expander's pin 1 (`SCL`) and pin 2 (`SDA`) wire straight to those balls on
both chips [W83601G DS p.4](#sources).

```{list-table} KGPE-D16 W83601G instances [AST2050-BMC-WIRING.md:353-354](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/AST2050-BMC-WIRING.md#L353-L354)
:header-rows: 1
:widths: 8 12 30 50

* - Ref
  - Bus
  - Address
  - Drives (LED nets)
* - `U27`
  - BMC I2C5
  - strap `A0/A1/A2` (pins 3/4/5)
  - 12 LEDs: `DIMM{A1,A2,B1,B2,C1,C2,D1,D2,E1,E2,F1,F2}ERRLED` — "write GP10–GP26 to light"
* - `U28`
  - BMC I2C5
  - strap `A0/A1/A2` (strapped differently from `U27`)
  - 4 LEDs: `DIMM{G1,G2,H1,H2}ERRLED`, remaining pins spare GPIO
```

The board wiring's "GP10–GP26" naming matches the datasheet's port naming
exactly (Port 1 = GP10–GP17, Port 2 = GP20–GP26; see §1.3), corroborating the
part identification. Each GP pin is an open-drain output able to sink **24 mA**
[W83601G DS p.3](#sources) — enough to sink an indicator LED directly.

Both chips' `RST#` (pin 19) is driven by the Super-I/O's resume reset: the
W83667HG-A's `GP57/RSMRST#` (pin 101, net `SIO_RSMRST#`) resets `U27`, `U28`,
the SP5100 and glue logic
[W83667HG-SUPERIO-WIRING.md:191](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/W83667HG-SUPERIO-WIRING.md#L191).

## 1.2 Addressing: the A0/A1/A2 straps

The SMBus address byte is a fixed prefix `0011` followed by the three strap
bits and the R/W bit [W83601G DS p.2](#sources):

```{list-table} W83601G address byte [W83601G DS p.2](#sources)
:header-rows: 1
:widths: 12 12 12 12 13 13 13 13

* - b7
  - b6
  - b5
  - b4
  - b3
  - b2
  - b1
  - b0
* - 0
  - 0
  - 1
  - 1
  - A2
  - A1
  - A0
  - R/W
```

so the 7-bit address range is **`0x18`–`0x1F`** (`0x18` + the A2:A1:A0 strap
value). The strap pins are the multi-function **GP20/A0 (pin 3), GP21/A1
(pin 4), GP22/A2 (pin 5)**: each is sampled as an address bit "during power-on
reset or RST# pin reset" and is otherwise an ordinary GP pin (CMOS I/O with an
internal pull-down in its input role) [W83601G DS p.4](#sources). The latched
strap value can be read back afterwards: CR0D (Port 2 input latched data) bits
2-0 hold A2–A0 [W83601G DS p.7](#sources). Sharing the strap pins with live
GPIO is safe because the sample happens only at reset — which on this board is
`SIO_RSMRST#` (§1.1), so the addresses re-latch on every resume reset.

`U27` and `U28` are strapped differently so both can share I2C5
[AST2050-BMC-WIRING.md:354](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/AST2050-BMC-WIRING.md#L354);
the *specific* strap values (hence which of `0x18`–`0x1F` each chip answers on)
have not yet been extracted from the schematic — an `i2cdetect` on the running
BMC's I2C5, or tracing the A0/A1/A2 nets, will pin them down.

## 1.3 Register map

Registers are selected by an 8-bit **index** (the SMBus command byte). Port 1
is the 8 pins GP10–GP17; Port 2 is the 7 pins GP20–GP26 (bit 7 of the Port-2
registers is reserved) [W83601G DS p.5-7](#sources).

```{admonition} Reset-default gotchas
:class: warning

- **Polarity inversion is NOT zero at reset**: CR02 defaults to `0xF0`
  (GP17–GP14 inverted) and CR0A to `0x70` (GP26–GP24 inverted)
  [W83601G DS p.5-7](#sources). Firmware that wants "register bit = pin level"
  semantics must clear both before use.
- CR14 bits 3-2 default to 0 = the port data/config registers are **not** reset
  by the `RST#` pin, only by internal power-on reset (set the bits to make
  `RST#` restore defaults) [W83601G DS p.8](#sources). The address straps *are*
  re-sampled on `RST#` (§1.2).
- **Chip ID discrepancy**: the datasheet's summary table lists the CR21 default
  as `0x12`, while the register description states `0x13` "for the W83601G"
  (with CR20 = `0x60`); revision 0.34 of the datasheet "updated CR21 Chip ID"
  [W83601G DS p.5,9,17](#sources). A read on real hardware settles it.
```

Other function bits, briefly [W83601G DS p.8-9](#sources):

- **CR14 (mode)** — bit 7 turns the GP/INT pin into the INT function; bit 6
  picks GP26 (pin 12) or GP17 (pin 18) as the INT pin; bit 5 selects pulse vs
  level INT output; bits 1-0 let `RST#` (not just power-on) latch the CR05/CR0D
  input snapshots.
- **CR15 (Power LED)** — bit 7 enables a hardware LED blinker that overrides
  the GP function on one selected pin (bit 3 picks the port, bits 2-0 the pin;
  GP16–GP10 or GP25–GP23 are selectable): bits 6-4 choose tri-state/driven-high,
  1 Hz, ½ Hz, ¼ Hz toggle (50 % duty), or driven low. A firmware could blink a
  DIMM error LED with zero further bus traffic.
- **Interrupts** — a transition on an enabled GP input sets the status register
  and asserts INT; reading CR10/CR11 clears both. In pulse mode the interrupt
  re-fires until every enabled status register has been read
  [W83601G DS p.11](#sources). (On the KGPE-D16 the chips drive LEDs, so the
  INT path is likely unused; no INT net has been traced.)

## 1.4 Access protocol

The interface is **SMBus 1.0 Write Byte / Read Byte only**
[W83601G DS p.10](#sources):

- **Write**: `START` — address byte (`0011 A2 A1 A0 0`) — index byte — data
  byte — `STOP`.
- **Read**: `START` — address byte (write) — pointer byte — repeated `START` —
  address byte (read) — data byte (master NACK) — `STOP`.

The bus is slow: minimum SCL clock period is 10 µs, i.e. **≤ 100 kHz**
[W83601G DS p.14](#sources). No block transfers, no auto-increment — one
register per transaction.

Lighting a DIMM error LED from a BMC shell is therefore a pair of plain
`i2cset` byte writes (via the
[Linux i2c-dev interface](https://docs.kernel.org/i2c/dev-interface.html)) —
configure the pins as outputs once, then set data bits:

```sh
# ADDR = 0x18 + A2:A1:A0 strap (per chip; see §1.2). BUS = the BMC's I2C5 adapter.
i2cset -y $BUS $ADDR 0x02 0x00   # CR02: clear Port 1 default polarity inversion
i2cset -y $BUS $ADDR 0x03 0x00   # CR03: Port 1 GP17-GP10 all outputs
i2cset -y $BUS $ADDR 0x01 0x01   # CR01: drive GP10
```

(Which data bit lights which `DIMM*ERRLED` net, and the drive polarity, are not
yet traced pin-by-pin — the wiring source records only the port-level mapping
of §1.1.)

## 1.6 Driver notes

There is **no mainline Linux driver**: `torvalds/linux` has no occurrence of
"w83601" in any tree (checked 2026-07-18 via GitHub code search) — unlike its
sibling hardware monitors (W83795G etc.), this part never grew a kernel
binding. That is no obstacle for an open BMC firmware: as §1.4 shows, bare
SMBus byte writes through `i2c-dev`/`i2cset` (or three writes from any I2C
master routine in U-Boot or a bare-metal loader) fully control the LEDs.
If kernel-level GPIO abstraction were ever wanted, the register model is close
to the PCA9555's input/output/polarity/configuration quartet ({doc}`pca9555`)
but with a flat one-register-per-index layout plus the latched-input,
pulse-output and LED-blink extras — a small bespoke `gpiochip` driver, not an
existing compatible. Prior art exists outside Linux: Wind River's VxWorks
PrPMC800 BSP ships a
[`w83601Gpio.h`](https://github.com/emuikernel/BDXDaq/blob/master/devel/VxWorks55/ppc/target/config/prpmc800/w83601Gpio.h)
register-definition header that matches the map in §1.3.
