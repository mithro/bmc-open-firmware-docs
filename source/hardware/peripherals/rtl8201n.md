# Realtek RTL8201N — 10/100 Ethernet PHY

The Realtek **RTL8201N-GR** (board reference **`U5`**, 64-pin QFN) is the
dedicated BMC management-LAN PHY on the ASUS KGPE-D16 ({doc}`/systems/kgpe-d16`).
It is a single-chip/single-port 10/100M Fast Ethernet PHYceiver with auto MDIX,
implementing the full 10Base-T / 100Base-TX physical layer (PCS, PMA, TP-PMD)
with a selectable **MII, RMII, or SNI** interface to the MAC.
[RTL8201N DS p.1–2](#sources) It bonds the AST2050's first FTGMAC100 MAC channel
to the physically separate BMC management RJ-45 — the out-of-band network port
that carries IPMI-over-LAN and the web/Redfish interface, independent of the
host's two Intel 82574L NICs ({doc}`intel-82574l`).

## Board wiring on the KGPE-D16

The AST2050 (`QU1`) reaches `U5` over the two-wire clause-22 management bus
(MDIO/MDC) plus the channel-1 data path. The data-path balls are the AST2050's
muxed MII/RMII pins; the board's net names (`AST_RMII1*`) and the SCU70
interface-mode strap dumped from running silicon (`SCU70[8:6] = 0b110` =
RMII MAC#1 + RMII MAC#2, {doc}`../registers/network-mac-phy` §11) show the
channel operating in **RMII** mode. TX-side nets carry an `R_` prefix — they run

Wiring notes:

- **Reset.** `AST_SRST#` is the *global* BMC-domain reset — it fans out from
  QU1 ball R20 to the JTAG header, the Super-I/O glue and `U5` pin 38
  (`PHYRSTB`), so the PHY resets (and re-latches its straps) together with the
  BMC. [QU1_pins.md "Power / reset"](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/pinmaps/QU1_pins.md#power--reset--platform-control-17)
- **50 MHz RMII reference clocks.** Both of the AST2050's RMII reference-clock
  inputs are driven by the clock generator `CU2` (ICS **ICS9112AM-16LFT**):
  net `C_MNG_50M_AST_RMII1RXCLK` → QU1 ball A7 (`MIITXCK/RMIIRCLK`, channel 1)
  and `C_MNG_50M_AST_RMII2RXCLK` → QU1 ball B7 (`MIIRXCK/RMII2RCLK`,
  channel 2).
  [QU1_pins.md "Ethernet RMII / NC-SI"](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/schematic-wiring/pinmaps/QU1_pins.md#ethernet-rmii--nc-si-18)
  The PHY itself is clocked from a 25 MHz crystal/oscillator on
  CKXTAL1/CKXTAL2 (pins 61/62). [RTL8201N DS Table 3 p.6](#sources)
- **Not the host NICs.** The AST2050's *second* channel (`AST_RMII2*`, NC-SI)
  runs to the two Intel 82574L host NICs (`LU1`/`LU2`) — that sideband path is
  a different device entirely; see {doc}`intel-82574l`. The RTL8201N serves only
  the dedicated management RJ-45. Full board-level context:
  {doc}`/systems/kgpe-d16-wiring`.

## MDIO access from the AST2050

The RTL8201N is a standard IEEE 802.3 **clause-22** managed PHY: frames of
32-bit preamble, `ST=01`, `OP` (read `10` / write `01`), 5-bit PHYAD, 5-bit
REGAD, turnaround, 16-bit data, clocked on MDC at up to 2.5 MHz.
[RTL8201N DS §8.1.2 p.16, Table 21](#sources) On the AST2050 these frames are
generated by the FTGMAC100 MAC's built-in MDIO controller: software writes
**PHYCR** (`MAC60`: `MIIRD`/`MIIWR` trigger bits, `PHYAD`, `REGAD`, MDC cycle
threshold) and transfers data through **PHYDATA** (`MAC64`) — the full
mechanism, including the driver polling sequence, is documented in
{doc}`../registers/network-mac-phy` (§7).

The PHY's 5-bit bus address is strapped on the `CONFIG[4:0]` pins (53, 54, 56,
57, 58), latched at power-on/reset; the latched value is readable back in
register 25 bits [11:7]. **Address `00000` must not be used — it puts the
RTL8201N into power-down mode**; valid addresses are `00001`–`11111` (up to 31
devices per MDIO bus). [RTL8201N DS Table 5 p.7, §8.1.2 p.16, Table 20 p.14](#sources)
(The datasheet is internally inconsistent about the strap pin list: Table 5
gives pins 53/54/56/57/58, §8.1.2 gives pins 58/57/56/54/52 — the strapped
address should be confirmed from register 25 on live hardware.)

  used in
  [`ftgmac100_26.c:1331-1332`](https://github.com/pkubaj/ast2050-linux-kernel/blob/master/drivers/net/ftgmac100_26.c#L1331-L1332).
  The project's Raptor analysis lists the RTL8201N in that driver's
  supported-PHY set —
  [RAPTOR_ENGINEERING_AST2050_ANALYSIS.md:492](https://github.com/mithro/ai-shenanigans-for-bmcs/blob/main/asus-kgpe-d16-firmware/RAPTOR_ENGINEERING_AST2050_ANALYSIS.md#L492).
- Mainline Linux's **e1000** driver (the RTL8201N is also used as the PHY on
  Intel CE4100 platforms) defines `RTL8201N_PHY_ID 0x8200` and treats it as a
  10/100-only PHY —
  [`e1000_hw.h:2903`](https://github.com/torvalds/linux/blob/master/drivers/net/ethernet/intel/e1000/e1000_hw.h#L2903),
  [`e1000_hw.c:1325-1327`](https://github.com/torvalds/linux/blob/master/drivers/net/ethernet/intel/e1000/e1000_hw.c#L1325-L1327).
- The mainline **Realtek phylib driver** has no RTL8201N entry by name, but its
  `PHY_ID_MATCH_EXACT(0x00008201)` entry — named "RTL8201CP Ethernet" — matches
  exactly the ID this chip reports (see the driver-binding note below).

### Registers 4–6 — auto-negotiation

- **Register 4 (ANAR)** — advertised abilities: defaults advertise
  100Base-TX FD/HD and 10Base-T FD/HD (bits 4.8:5 = 1), selector `00001`
  (CSMA/CD). Bits 4.11 `TXFC` / 4.10 `RXFC` advertise flow control — the MAC
  sets bit 4.10 over MDIO to add pause ability to the NWay advertisement.
  [RTL8201N DS Table 13 p.11, §8.3 p.19](#sources)
- **Register 5 (ANLPAR)** — read-only image of the link partner's ability word
  (same bit layout; 5.14 = partner Acknowledge). Bits 5.7/5.5 are also set when
  the link is established by parallel detection. [RTL8201N DS Table 14 p.12](#sources)
- **Register 6 (ANER)** — expansion status: 6.4 `MLF` multiple-link-fault,
  6.3 `LP_NP_ABLE`, 6.2 `NP_ABLE` (internal use), 6.1 `PAGE_RX`,
  6.0 `LP_NW_ABLE`. [RTL8201N DS Table 15 p.13](#sources)

## Strap (CONFIG) pins

All operating-mode defaults are latched from pins at power-on/reset (the pins
revert to their run-time functions afterwards); most can then be overridden
over MDIO. [RTL8201N DS Table 5 p.7, §8.4 p.19](#sources)

```{list-table} Device configuration straps
:header-rows: 1
:widths: 18 10 72

* - Strap
  - Pin
  - Function
* - CONFIG[4:0]
  - 53, 54, 56, 57, 58
  - PHY (MDIO) address; `00000` = power-down mode, do not use
* - CONFIG[5] (CRS)
  - 47
  - High = repeater mode (default NIC/switch mode, pulled low)
* - CONFIG[6] (COL)
  - 46
  - MII/SNI select: low = SNI (10 Mbps only), high = MII (default)
* - CONFIG[7] (ANE)
  - 37
  - High = auto-negotiation enabled; low = forced mode from SPEED/DUPLEX
* - CONFIG[8] (RXER)
  - 35
  - High = 100Base-FX fiber mode (5.1 kΩ pull-up); weak internal pull-down defaults to UTP
* - CONFIG[9] (SPEED)
  - 33
  - With ANE low: high = force 100 Mbps, low = force 10 Mbps; with ANE high, sets the advertised ability
* - DUPLEX
  - 51
  - With ANE low: high = full duplex; with ANE high, sets the advertised ability
* - RXDV/RMII
  - 16

  - 52
```

## RTL8201N vs RTL8201CP

Both are Realtek single-port 10/100 clause-22 PHYs reporting the same
`0x0000`/`0x8201` identifier signature, with a near-identical vendor register
set (16 NSR / 17 LBREMR / 18 REC / 19 SNR). Differences that matter for this
board and for driver/model work:

```{list-table} RTL8201N vs RTL8201CP
:header-rows: 1
:widths: 24 38 38

* - Aspect
  - RTL8201N (KGPE-D16, `U5`)
  - RTL8201CP (C410X-era reference, §10 of the MAC page)
* - Package
  - 64-pin QFN, pin-to-pin compatible with the RTL8211B
  - 48-pin part (see its datasheet)
* - RMII selection
  - Strap (pin 16) **or** writeable register bit 0.0
  - Strap only; register 17.0 `RMIIMODE` is a read-only indicator
* - Register 17.0
  - `SNIBMODE` (SNI-mode indicator)
  - `RMIIMODE` (RMII-mode indicator)
* - Register 25
  - Documented: PHYAD readback + LINK10/LINK100
  - Not documented (vendor set stops at register 19)
* - SNI (7-wire serial) interface
  - Yes (10 Mbps only)
  - Not documented
* - Supply / process
  - 3.3 V in, internal 1.5 V/1.8 V regulators, 0.15 µm
  - 3.3 V single supply
* - PHY ID (regs 2/3)
  - `0x0000` / `0x8201`
  - `0x0000` / `0x8201`
```

RTL8201N column: [RTL8201N DS p.2, p.9–14, §8.10 p.23](#sources); RTL8201CP
column: [RTL8201CP DS §6 p.8–13](#sources) and the tabulated version in
{doc}`../registers/network-mac-phy` (§10). Because the IDs are identical,
**software cannot distinguish the two by the identifier registers alone** — a
QEMU PHY model of either part will satisfy an ID-probing driver for both.

```{admonition} Mainline phylib + the Raptor vendor driver
:class: note

- **Mainline**: the Realtek PHY driver now lives at
  [`drivers/net/phy/realtek/realtek_main.c`](https://github.com/torvalds/linux/blob/master/drivers/net/phy/realtek/realtek_main.c)
  (the old single-file `drivers/net/phy/realtek.c` path no longer exists). Its
  entry
  [`PHY_ID_MATCH_EXACT(0x00008201)` — "RTL8201CP Ethernet"](https://github.com/torvalds/linux/blob/master/drivers/net/phy/realtek/realtek_main.c#L2689-L2694)
  matches exactly what the RTL8201N reports, so **the RTL8201N binds to the
  RTL8201CP entry**. That entry supplies only page-access helpers — link
  management is plain `genphy` (BMCR/BMSR/ANAR), which is all this PHY needs.
  An RTL8201N silicon revision other than 1 (PHYID2 ≠ `0x8201` in the low
  nibble) would miss the exact match and fall back to the "Generic PHY" driver
  — functionally equivalent for 10/100 auto-negotiation
  ([Linux PHY subsystem](https://docs.kernel.org/networking/phy.html)).
- **Raptor 2.6.28 vendor driver**: `ftgmac100_26` matches the RTL8201N
  explicitly (`PHYID_RTL8201N 0x00008200`, revision-masked) and treats it as a
  10/100 PHY with standard BMCR/BMSR handling —
  [`ftgmac100_26.h:461-474`](https://github.com/pkubaj/ast2050-linux-kernel/blob/master/drivers/net/ftgmac100_26.h#L461-L474).
- **On the modern KGPE-D16 port** the PHY side is unproblematic; the open
  issue is on the MAC side (the mainline `ftgmac100` RMII/`phy-mode` handling —
  see the note in {doc}`../registers/network-mac-phy` §7 and {doc}`/drivers/linux`).
```
